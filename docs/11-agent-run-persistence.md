## 1. Scope

This milestone adds the smallest production-shaped persistence slice for:

```text
create AgentJob
→ atomically start AgentRun as RUNNING
→ execute the existing in-memory orchestrator (unchanged)
→ transactionally persist the full ordered trace
→ persist exactly one terminal outcome
→ read the job, run, trace, and outcome back
```

It does **not** add a queue, a claim-worker, execution-token fencing, cancellation, a maintenance sweep, live progress streaming, an API layer, or a UI. `docs/03-technical-design.md` §16 and `docs/04-agent-design.md` §16 describe a larger *future* production design with all of that; this document describes what is actually built today, and is explicitly a simpler precursor to that design — not an implementation of it. See §9 for exactly how they relate.

The API layer this document explicitly excludes is now built — see `docs/12-agent-run-api.md` for `apps/api`, the local-only synchronous NestJS API that calls `createAgentRunService`/`createPrismaAgentRunRepository` from `@opspilot/agent-runtime` exactly as `apps/worker`'s persisted demo does, over the same repository functions and schema described below.

## 3. Prisma 7 Configuration

Pinned exactly, no caret: `prisma` / `@prisma/client` / `@prisma/adapter-pg` = `7.9.0`, `pg` = `8.22.0`, `@types/pg` = `8.20.0`.

Prisma 7 removed the bundled Rust query-engine binary — every connection requires an explicit driver adapter. `packages/database/src/client.ts`'s `createPrismaClient()` builds a `pg.Pool`, wraps it in `@prisma/adapter-pg`'s `PrismaPg`, and constructs `PrismaClient({ adapter })`. Callers own the pool's lifecycle: `createPrismaClient()` returns `{ prisma, close() }`, and `close()` must be called (`prisma.$disconnect()` then `pool.end()`) or the process hangs on an open connection. The persisted demo and every integration test call `close()`/`.close()` in a `finally`/`afterAll` block.

`schema.prisma`:
```prisma
generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma-client"
}

datasource db {
  provider = "postgresql"
}
```
No `url` in the datasource block — the generator is `prisma-client` (the Prisma 7 driver-adapter-native generator, not the retired `prisma-client-js`), with an explicit `output` path (pnpm's non-hoisted `node_modules` makes implicit generation unreliable across workspace packages).

`prisma.config.ts`:
```ts
export default defineConfig({
  schema: path.join(import.meta.dirname, "prisma", "schema.prisma"),
  migrations: { path: path.join(import.meta.dirname, "prisma", "migrations") },
  datasource: {
    url: process.env.DATABASE_URL ?? "",
    shadowDatabaseUrl: process.env.SHADOW_DATABASE_URL ?? "",
  },
});
```
Both fall back to `""` rather than throwing — `prisma generate` and `tsc` must succeed with no `.env` file and no running PostgreSQL at all (verified: `pnpm install && pnpm --filter @opspilot/database run db:generate && pnpm -r run typecheck && pnpm -r run test` all pass on a completely fresh clone).

**A name collision to be aware of:** the generated `prisma-client` output re-exports a type literally named `AgentTraceEvent` (aliasing `Prisma.AgentTraceEventModel`, the raw DB row shape) — colliding with this project's own `AgentTraceEvent` domain type from `@opspilot/contracts`. Nothing in this codebase imports that name from the generated client (only `PrismaClient` and the `Prisma` namespace are imported from it); this is called out explicitly so a future contributor doesn't accidentally `import type { AgentTraceEvent } from "../generated/prisma-client/client"` and get the wrong type.

**Deviation from the approved plan, discovered only by running the real CLI:** `prisma migrate diff --to-config-datasource` requires its `shadowDatabaseUrl` target to already exist — unlike `prisma migrate dev`, which auto-creates and drops its own shadow database. `scripts/ensure-test-database.sh` was extended to also idempotently create `opspilot_shadow` (see §8).

**Local port deviation:** the developer machine this was implemented on already had something bound to the default port 5432. `docker-compose.yml` maps host `55432` -> container `5432` instead; `.env.example` reflects this. The container's internal port is unaffected.

## 4. Schema

Three tables. No `PENDING` status, no generated columns, no separate outcome table.

### `agent_jobs`

| Column | Type | Notes |
|---|---|---|
| `id` | `UUID` PK | `gen_random_uuid()` default |
| `ticket_context` | `JSONB NOT NULL` | Verbatim `{ ticketId, summary }` snapshot |
| `external_ticket_id` | `TEXT NOT NULL` | Written explicitly by the mapper, kept consistent with `ticket_context` by a CHECK constraint (not a generated column) |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | |

No `status` column, no `latest_run_id` pointer — there is no queue/claim mechanic in this milestone for a stored job status to track; "the current run" is just the run with the highest `attempt_number`.

### `agent_runs`

| Column | Type | Notes |
|---|---|---|
| `id` | `UUID` PK | |
| `job_id` | `UUID NOT NULL` FK -> `agent_jobs.id` | `ON DELETE CASCADE` |
| `attempt_number` | `INTEGER NOT NULL` | `UNIQUE(job_id, attempt_number)` |
| `status` | `TEXT NOT NULL` | `RUNNING \| COMPLETED \| FAILED` — no `PENDING` |
| `provider_mode` | `TEXT NOT NULL` | `FAKE \| LIVE` |
| `model_identifier` | `TEXT NULL` | |
| `report` | `JSONB NULL` | Set only on `COMPLETED` |
| `failure_code` | `TEXT NULL` | Set only on `FAILED`; one of the 10 `AgentOrchestratorErrorCode` values — **never a free-text message** (§6) |
| `started_at` | `TIMESTAMPTZ NOT NULL` | Non-nullable — every row is created already `RUNNING` |
| `finished_at` | `TIMESTAMPTZ NULL` | Set only on a terminal status |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | |

### `agent_trace_events`

| Column | Type | Notes |
|---|---|---|
| `id` | `UUID` PK | |
| `run_id` | `UUID NOT NULL` FK -> `agent_runs.id` | `ON DELETE CASCADE` |
| `sequence_number` | `INTEGER NOT NULL` | `UNIQUE(run_id, sequence_number)`, 1-based, contiguous |
| `event_type` | `TEXT NOT NULL` | One of the 4 real `AgentTraceEvent` variants |
| `payload` | `JSONB NOT NULL` | The verbatim `AgentTraceEvent` object |
| `created_at` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | |

Sequence numbers are the 1-based index of each event in the orchestrator's returned `trace` array — safe because there is exactly one synchronous writer per run (the whole array is inserted in one transaction, once, after the orchestrator completes). This is why no `nextStepSequence`-style atomic counter is needed here, unlike the future `AgentStep` design (§9), which must serialize multiple concurrent writers.

### Named CHECK constraints (all 12, exact names — hand-authored, not Prisma-generated)

```text
agent_jobs_ticket_context_is_object_chk       jsonb_typeof(ticket_context) = 'object'
agent_jobs_external_ticket_id_matches_chk     external_ticket_id = ticket_context ->> 'ticketId' AND <> ''

agent_runs_attempt_number_chk                 attempt_number >= 1
agent_runs_status_chk                         status IN ('RUNNING','COMPLETED','FAILED')
agent_runs_provider_mode_chk                  provider_mode IN ('FAKE','LIVE')
agent_runs_report_is_object_chk               report IS NULL OR jsonb_typeof(report) = 'object'
agent_runs_failure_code_chk                   failure_code IS NULL OR failure_code IN (<10 exact codes>)
agent_runs_terminal_outcome_chk               see below

agent_trace_events_sequence_number_chk        sequence_number >= 1
agent_trace_events_payload_is_object_chk      jsonb_typeof(payload) = 'object'
agent_trace_events_event_type_chk             event_type IN (<4 exact variants>)
agent_trace_events_event_type_matches_chk     event_type = payload ->> 'type'
```

Terminal-outcome invariant (`agent_runs_terminal_outcome_chk`):
```sql
(status = 'RUNNING'   AND finished_at IS NULL     AND report IS NULL     AND failure_code IS NULL) OR
(status = 'COMPLETED' AND finished_at IS NOT NULL AND report IS NOT NULL AND failure_code IS NULL) OR
(status = 'FAILED'    AND finished_at IS NOT NULL AND report IS NULL     AND failure_code IS NOT NULL)
```
Enforced by Postgres itself — any `UPDATE` that bypasses the repository and tries to set an illegal combination is rejected, independent of application code (integration-tested by attempting exactly that via raw SQL).

**Verification, kept deliberately separate:**
- `pnpm db:migrate:status` — migration *history* only (pending/missing/modified/failed migrations vs. `_prisma_migrations`). Not a schema-drift check.
- `pnpm db:migrate:drift` — `prisma migrate diff --from-migrations prisma/migrations --to-config-datasource --exit-code` (exit `0` no difference, `1` command error, `2` difference found). **Cannot see the CHECK constraints above** — Prisma's schema DSL doesn't model them, so a dropped or hand-edited constraint would not show up here.
- `packages/database/src/schema-constraints.integration.test.ts` — the actual authority: queries `pg_constraint` directly and asserts all 12 constraints exist by exact name.

## 5. Transactions

Lock order: `AgentJob` -> `AgentRun` -> `AgentTraceEvent`, consistent everywhere. `agent_jobs` has no mutable state in this milestone (no claim/lease/token fields), so only `startRun` locks it; finalize transactions lock only the target `AgentRun` row.

### `startRun` — one atomic transaction, no `PENDING`, the sole source of a run's ticket context

```text
BEGIN
  SELECT id, ticket_context, external_ticket_id, created_at FROM agent_jobs WHERE id = $jobId FOR UPDATE
  -- not found -> PERSISTENCE_NOT_FOUND
  -- runtime-validate + map the selected row (fromAgentJobRow) BEFORE inserting AgentRun —
  -- a malformed stored snapshot throws here, rolling back the transaction with no AgentRun created
  SELECT COALESCE(MAX(attempt_number), 0) FROM agent_runs WHERE job_id = $jobId  -- safe: serialized by the lock above
  INSERT agent_runs (job_id, attempt_number, status='RUNNING', started_at=now(), provider_mode, model_identifier)
  -- returns { job: <validated AgentJobRecord from the locked row>, run: <new AgentRunRecord> }
COMMIT
```
`startRun` selects the **complete** `agent_jobs` row (`id`, `ticket_context`, `external_ticket_id`, `created_at`) under the same `FOR UPDATE` lock used to allocate `attempt_number` — not just a bare existence check — and returns it alongside the new `AgentRun` as `StartedAgentRun { job: AgentJobRecord; run: AgentRunRecord }`. This is deliberate: `AgentJobRecord` is a public structural TypeScript interface, and accepting one from a caller (as an earlier revision of this service did) would let a caller combine one job's `id` with a different job's `ticketContext` — storing the resulting `AgentRun` under job A while investigating job B's ticket. Because the job snapshot and the `AgentRun.jobId` foreign key now both originate from this single locked read, that combination is structurally impossible: `AgentRunService.executeAndPersist` (§7) accepts only `jobId: string` and derives everything else from what `startRun` actually read out of PostgreSQL.

Concurrent `startRun` calls for the same job serialize on the `AgentJob` lock and always allocate unique, increasing `attempt_number`s (integration-tested with two real concurrent Prisma clients). `createRun`+`markRunRunning` from an earlier draft of this design were collapsed into this single transaction — a separately-committed `PENDING` row would only have added a crash window with nothing in this milestone to observe or act on it.

A crash after this commits but before finalization leaves the row permanently `RUNNING` — an accepted, explicitly documented gap (§10); there is no reaper this milestone.

### `finalizeCompleted` / `finalizeFailed` — exact-replay-safe

```text
1. Runtime-validate the trace and terminal outcome BEFORE opening a transaction (§6) — an
   invalid write never touches the database.
2. BEGIN
3. SELECT status FROM agent_runs WHERE id = $runId FOR UPDATE
   -- not found -> PERSISTENCE_NOT_FOUND
4. IF status = 'RUNNING':
     INSERT the full trace batch (sequence_number = 1..N)
     UPDATE agent_runs SET status = 'COMPLETED'|'FAILED', report|failure_code, finished_at = now()
     COMMIT  -- happy path, one atomic write
5. ELSE (already terminal):
     Compare the stored trace to the incoming trace as ordered {sequenceNumber, payload}
     pairs — jsonb_agg(jsonb_build_object('sequenceNumber', sequence_number, 'payload',
     payload) ORDER BY sequence_number) — via PostgreSQL JSONB equality, not JS string
     comparison. Comparing payload alone would incorrectly accept a stored [1, 3] as a
     replay of an incoming [1, 2] whenever the payloads happen to be in the same order;
     including sequenceNumber in the compared value closes that gap. Array element order
     is preserved by JSONB equality (it matters); object key order is normalized away
     (it doesn't).
     Compare stored report/failure_code to the incoming outcome via IS NOT DISTINCT FROM.
     IF status matches AND trace matches AND outcome matches:
       return the existing row, unchanged — idempotent success, zero new rows inserted
     ELSE:
       PERSISTENCE_CONFLICT
COMMIT
```
A completed-vs-failed race (two concurrent finalize calls) always resolves to exactly one winner: the losing transaction's `FOR UPDATE` blocks until the first commits, then observes non-`RUNNING` status and falls into the replay-or-conflict branch, where a different terminal type never satisfies the idempotent-match condition.

**Real transactional rollback, not just pre-transaction validation.** Two distinct integration tests exist and must not be conflated: "rejects an invalid report before any transaction begins, leaving the run and trace untouched" proves that runtime validation (step 1 above) rejects a bad write before any transaction opens — there is nothing to roll back. A separate test, "a real PostgreSQL failure after the trace insert rolls back the entire transaction, leaving the run RUNNING with zero trace rows", proves genuine mid-transaction atomicity: a test-only trigger (created and dropped entirely within that one test, never a production hook) makes the terminal `UPDATE` fail *after* the trace `createMany` has already executed inside the same transaction, and asserts both writes are rolled back together.

### `getAgentRun`

One `RepeatableRead`-isolated transaction reads the run+job (with the job included) and the ordered trace (`ORDER BY sequence_number ASC`, never re-sorted in application code), revalidates every JSON field (§6) — including that the sequence is exactly contiguous 1..N (a gap, duplicate, zero, or non-one-based start throws `PERSISTENCE_VALIDATION_FAILED`) — and assembles `PersistedAgentRun`.

## 6. Runtime Validation Boundary

TypeScript types are never trusted alone. Every value crossing into or out of the database is validated (or revalidated) against a Zod schema from `@opspilot/contracts` (or the narrow `TicketContextSchema` in `packages/database/src/validation.ts`, since no shared ticket-context contract existed before this milestone):

- **Write**: `createJob`'s ticket context, every trace event in a finalize batch, the completed report, the failure code.
- **Read**: `ticket_context`, every ordered trace payload, the report, the failure code — before `getAgentRun` returns them.

A validation failure maps to `PersistenceError("PERSISTENCE_VALIDATION_FAILED", "<fixed message>", { cause })` — the Zod issue array is retained only as the internal `cause`, never surfaced in `.message`. This catches what the CHECK constraints structurally cannot: e.g. `agent_trace_events_event_type_matches_chk` only verifies `event_type` matches `payload->>'type'`, not that the payload has the right *shape* for that type (a `TOOL_REQUESTED` payload missing `toolCallId` passes the constraint but fails Zod revalidation on read).

**Trace contiguity is also validated on read**, not only assumed: `fromTraceEventRows` (`packages/database/src/mappers.ts`) checks that the ordered rows' `sequenceNumber`s are exactly `1, 2, ..., N` (matching array position) before revalidating payloads — a gap, duplicate, zero, or non-one-based starting sequence throws `PERSISTENCE_VALIDATION_FAILED` with a fixed message, since such a row could only exist through a path other than `finalizeCompleted`/`finalizeFailed` (which always assign contiguous sequence numbers), e.g. a manual `INSERT`.

**Failure data is narrowed to a stable code only.** `failure_message` is not a column. `packages/database/src/failure-messages.ts` exports a fixed, exhaustive `Record<AgentOrchestratorErrorCode, string>` (tested for exact parity with `AgentOrchestratorErrorCodeSchema.options` — no missing, no extra key); `getAgentRun`'s `FAILED` outcome derives its `message` from this map at read time. `finalizeFailed`'s signature accepts only a `code` parameter — there is no `message` parameter for a raw orchestrator/exception string to flow through, by construction.

## 7. Error Taxonomy

`PersistenceErrorCode`: `PERSISTENCE_UNAVAILABLE | PERSISTENCE_CONFLICT | PERSISTENCE_VALIDATION_FAILED | PERSISTENCE_NOT_FOUND`. `normalizeDatabaseError` (`packages/database/src/errors.ts`) maps Prisma/driver error codes (`P2002`/`23505` -> conflict, `23514` -> conflict, `P2025` -> not found, `P1001`/`ECONNREFUSED`/etc. -> unavailable, everything else -> unavailable) to one of these, with a fixed message — never the raw Prisma message, connection string, or SQL text.

`AgentRunServiceErrorCode` (`apps/worker/src/persistence/agent-run-service-error.ts`): `AGENT_EXECUTION_CRASHED` only — thrown, not returned, when `runAgentOrchestrator` itself throws an unexpected exception after `startRun` has already committed a `RUNNING` row. Fixed message ("The agent execution terminated unexpectedly."), the raw cause retained only via `Error.cause` for internal debugging, `runId` attached so the caller can log which row is now orphaned. Never converted to a `PersistenceError`, never converted to an `AgentOrchestratorErrorCode`, never written to the database as a `FAILED` outcome.

`ExecuteAndPersistResult` (the service's return type for the two *expected* outcomes):
```ts
type ExecuteAndPersistResult =
  | { persistence: "persisted"; run: PersistedAgentRun }
  | { persistence: "unavailable"; stage: "run-creation"; error: PersistenceError }
  | { persistence: "unavailable"; stage: "finalization"; runId: string; agentResult: AgentOrchestratorResult; error: PersistenceError };
```
`retryFinalization(runId, agentResult)` is the explicit retry path for the `finalization` stage — see §10 for its exact durability limit.

**Callers supply only `jobId` — never a job object, never a ticket context.** `ExecuteAndPersistParams` extends `Omit<AgentOrchestratorParams, "initialConversation">` and additionally requires `jobId: string`, not `job: AgentJobRecord`. Accepting a caller-constructed `AgentJobRecord` (an earlier revision of this service did exactly that) is not actually safe: `AgentJobRecord` is a public structural TypeScript interface, so a caller could construct one containing job A's `id` with job B's `ticketContext`, silently investigating the wrong ticket under job A's `AgentRun`. `executeAndPersist` instead calls `repository.startRun(params.jobId, ...)`, which returns a `StartedAgentRun { job: AgentJobRecord; run: AgentRunRecord }` — the job snapshot `startRun` itself just read from PostgreSQL under its own `FOR UPDATE` lock (§5) — and builds the sole `ticket_context` conversation message from `started.job.ticketContext.ticketId`/`.summary`. The ticket context and the `AgentRun.jobId` foreign key therefore always originate from the exact same locked database row; there is no path by which they can disagree.

## 8. Test Database Lifecycle

One Postgres service (`docker-compose.yml`, `postgres:16-alpine`, host port `55432` — see §3), three logical databases: `opspilot` (dev), `opspilot_test` (integration tests), `opspilot_shadow` (throwaway comparison target for `db:migrate:drift` only). `docker/postgres-init/001-create-test-db.sql` creates all on a genuinely fresh volume as a convenience; `scripts/ensure-test-database.sh` is the authoritative, idempotent mechanism, safe to rerun on any volume state.

```text
pnpm infra:up            docker compose up -d
pnpm infra:down          docker compose down (keeps data)
pnpm infra:down:reset    docker compose down -v (deletes all local DB data)
pnpm db:test:ensure      idempotently ensures opspilot_test and opspilot_shadow exist
pnpm db:migrate:deploy   applies committed migrations to opspilot (DATABASE_URL)
pnpm db:migrate:test     applies committed migrations to opspilot_test (TEST_DATABASE_URL remapped to DATABASE_URL for the child process — genuinely remapped, not just documented)
pnpm db:migrate:status   migration-history check
pnpm db:migrate:drift    supported-schema drift check (see §4)
pnpm db:reset:dev / db:reset:test   prisma migrate reset --force against each database
```

Integration tests (`packages/database/**/*.integration.test.ts`) run serially (`fileParallelism: false`, `vitest.integration.config.ts`) against `opspilot_test`, truncating all three tables in `afterEach`. If PostgreSQL is unreachable, the suite fails loudly with one concise setup message (`packages/database/src/test/test-db.ts`) — never a silent skip.

## 9. Relationship to the Future Design (docs/03 §16, docs/04 §16)

This milestone is a deliberately simpler precursor, not a partial implementation:

| | This milestone | Future design (docs/03/04 §16) |
|---|---|---|
| Job creation | No queue; a run is created directly `RUNNING` | `AgentJob` is a queue-claim row; a worker polls `PENDING` rows via `FOR UPDATE SKIP LOCKED` |
| Run states | `RUNNING \| COMPLETED \| FAILED` | `PENDING \| RUNNING \| COMPLETED \| FAILED \| CANCELLED` |
| Trace table | `agent_trace_events`, 4 event types, no counter column | `AgentStep`, larger vocabulary (`RUN_QUEUED`, `RUN_STARTED`, `CLASSIFICATION`, ...), `AgentRun.nextStepSequence` atomic counter |
| Writers per run | Exactly one, synchronous | Multiple concurrent (worker, maintenance sweep, approval API) |
| Crash recovery | None — orphaned `RUNNING` rows are an accepted gap | 60-second maintenance sweep marks expired claims `FAILED` |

The table/type names are kept deliberately distinct (`agent_trace_events` vs. the future `AgentStep`) specifically so this milestone is never mistaken for an implementation of that richer design.

## 10. Current Limitations (accepted, explicitly out of scope this milestone)

- **No reaper/recovery for orphaned `RUNNING` rows.** A crash between `startRun` committing and finalization (including an `AGENT_EXECUTION_CRASHED` throw) leaves the row `RUNNING` forever, until a future reaper/recovery milestone.
- **`retryFinalization` is not process-restart-safe.** It is caller-controlled, in-memory-only retry, valid only while the calling process still holds the original `AgentOrchestratorResult`. It handles a failed finalization call and an uncertain post-commit connection failure (via the exact-replay contract, §5) — it does **not** handle process restart, loss of the in-memory result, or durable resumption of any kind.
- No queue, no scheduling, no automatic retries, no cancellation, no execution-token fencing, no multiple concurrent workers, no live progress streaming, no API layer, no UI, no deployment, no CI wiring, no pgvector/embeddings/runbook persistence, no `idempotency_key` on `AgentJob`/`AgentRun` (no HTTP layer exists yet to need one).
